# 1️⃣1️⃣ Subqueries (Nested Queries)

## 📖 What is a Subquery?

A **Subquery**, also known as a **Nested Query** or **Inner Query**, is a SQL query written inside another SQL query.

The inner query executes first, and its result is passed to the outer query.

```
Outer Query
     │
     ▼
Inner Query (Subquery)
     │
Returns Result
     │
     ▼
Outer Query Executes
```

A subquery can be used in almost every SQL statement, including:

- SELECT
- INSERT
- UPDATE
- DELETE
- WHERE
- HAVING
- FROM

---

## Why Do We Use Subqueries?

Imagine a company wants to find employees who earn more than the average salary.

Instead of manually calculating the average salary and then writing another query, SQL allows us to perform both operations together using a subquery.

Subqueries are widely used in:

- ETL Pipelines
- Data Validation
- Reporting
- Data Warehousing
- Business Intelligence
- Analytics

---

Operators used

```

ANY
```

Example


## 3. Multiple Column Subquery

Returns multiple columns.

```sql
SELECT *
FROM Employee
WHERE (Dept_ID,Salary)
IN
(
SELECT Dept_ID,MAX(Salary)
FROM Employee
GROUP BY Dept_ID
);
```

---

Returns exactly one value.

Example


# Operators Used with Subqueries

## IN

Checks multiple values.

```sql
WHERE Dept_ID IN
(
SELECT Dept_ID
FROM Department
);
```

---

## EXISTS

Returns TRUE if subquery returns at least one row.

```sql
SELECT *
FROM Department d
WHERE EXISTS
(
SELECT *
FROM Employee e
WHERE e.Dept_ID=d.Dept_ID
);
```

---

## NOT EXISTS

Returns rows when subquery returns nothing.

```sql
SELECT *
FROM Department d
WHERE NOT EXISTS
(
SELECT *
FROM Employee e
WHERE e.Dept_ID=d.Dept_ID
);
```

---

## ANY

Returns TRUE if condition matches any row.

```sql
SELECT *
FROM Employee
WHERE Salary >
ANY
(
SELECT Salary
FROM Employee
WHERE Dept_ID=1
);
```

---

## ALL

Condition must satisfy every row.

```sql
SELECT *
FROM Employee
WHERE Salary >
ALL
(
SELECT Salary
FROM Employee
WHERE Dept_ID=2
);
```

# Correlated Subquery

## Definition

A **Correlated Subquery** is a subquery that depends on the outer query.

Unlike a normal subquery, it **cannot execute independently**.

For every row of the outer query, the correlated subquery executes again.

```
Outer Row 1
      │
Subquery Executes

Outer Row 2
      │
Subquery Executes

Outer Row 3
      │
Subquery Executes
```

This is why correlated subqueries are usually slower than normal subqueries.

---

# Syntax

```sql
SELECT *
FROM table1 t1
WHERE column Operator
(
SELECT value
FROM table2 t2
WHERE t1.column=t2.column
);
```

---

# Example

Find employees earning above the average salary of their own department.

```sql
SELECT
e1.Name,
e1.Department,
e1.Salary
FROM Employee e1
WHERE Salary >
(
SELECT AVG(e2.Salary)
FROM Employee e2
WHERE e1.Department=e2.Department
);
```

# Normal Subquery vs Correlated Subquery

| Feature | Subquery | Correlated Subquery |
|----------|----------|--------------------|
|Execution|Runs Once|Runs for Every Row|
|Independent|Yes|No|
|Performance|Fast|Slower|
|Uses Outer Query|No|Yes|
|Common Usage|Filtering|Row-wise Comparison|

---

# Advantages

### Subquery

- Easy to understand
- Cleaner SQL
- Good for filtering
- Reusable logic

### Correlated Subquery

- Performs row-by-row comparison
- Useful for departmental analysis
- Excellent for EXISTS and NOT EXISTS
- Solves complex business problems

---

# Limitations

### Subquery

- Can become deeply nested
- Harder to debug
- May reduce readability

### Correlated Subquery

- Executes repeatedly
- Slower on large datasets
- Often replaced by JOINs or Window Functions for better performance

---

# Real-World Data Engineering Use Cases

- Find duplicate records
- Validate ETL data
- Compare source and target tables
- Detect missing records
- Incremental data loading
- Data reconciliation
- Identify highest sales per region
- Department-wise salary analysis
- Data quality checks
- Slowly Changing Dimension (SCD) validation

---

# Best Practices

- Prefer simple subqueries when possible.
- Avoid excessive nesting.
- Use aliases (`e1`, `e2`) for readability.
- Replace correlated subqueries with JOINs or Window Functions if performance is critical.
- Use `EXISTS` instead of `IN` for large datasets when appropriate.
- Analyze execution plans for complex queries.
