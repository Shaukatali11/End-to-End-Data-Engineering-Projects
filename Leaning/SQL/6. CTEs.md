# 🟠 Chapter 3.3A – Common Table Expressions (CTEs)

> **"Write complex SQL queries in a simple, readable, and maintainable way."**

---

# 📖 What is a Common Table Expression (CTE)?

A **Common Table Expression (CTE)** is a **temporary named result set** that exists **only during the execution of a SQL statement**.

It is defined using the `WITH` keyword and can be referenced like a normal table within the same query.

Think of a CTE as creating a **temporary virtual table** that helps break a complex SQL query into smaller, readable, and reusable parts.

Unlike a permanent table, a CTE:

- Doesn't consume permanent database storage.
- Exists only while the query is executing.
- Cannot be accessed after the query finishes.
- Makes SQL much easier to understand.

---

## Simple Definition

> **A CTE is a temporary result set created using the `WITH` clause that simplifies complex SQL queries.**

---

# 🤔 Why were CTEs introduced?

Before CTEs, developers mainly relied on:

- Nested Subqueries
- Derived Tables
- Temporary Tables

As SQL queries became more complex, they became difficult to read and maintain.

Example without CTE:

```sql
SELECT *
FROM
(
    SELECT *
    FROM
    (
        SELECT *
        FROM Employee
    ) A
) B;
```

Imagine a query with **5–10 nested subqueries**. Understanding or debugging it becomes very difficult.

Microsoft SQL Server introduced **Common Table Expressions** to solve this readability and maintainability problem.

---

# Problems Before CTEs

### ❌ Poor Readability

```sql
SELECT *
FROM
(
    SELECT *
    FROM
    (
        SELECT *
        FROM Orders
    ) X
) Y;
```

Hard to understand.

---

### ❌ Difficult Debugging

Finding which subquery is causing an error becomes challenging.

---

### ❌ Code Repetition

The same subquery may need to be written multiple times.

---

### ❌ Difficult Maintenance

If business logic changes, every repeated subquery must be updated.

---

# Why Data Engineers Love CTEs ❤️

Suppose you're building an ETL pipeline.

```
Raw Data
     │
Cleaning
     │
Filtering
     │
Aggregation
     │
Transformation
     │
Final Dataset
```

Each transformation can be represented as a separate CTE, making the SQL resemble the actual ETL process.

Instead of one huge query, you get several logical steps.

---

✅ Better readability

✅ Easier debugging

✅ Reusable logic

✅ Modular SQL

✅ Cleaner ETL pipelines

✅ Easier collaboration

✅ Simplifies complex transformations

Basic syntax:

```sql

```

# Understanding the Flow

SQL executes the query like this:

```
Step 1

WITH HighSalaryEmployee

↓

Runs First

↓

Temporary Result Created

↓

Main Query Executes

↓

Output Returned

↓

CTE Disappears
```

The CTE is **not stored permanently**.

Query

# Example 2 – CTE with Aggregate Function

Find departments having more than two employees.

```sql
WITH DepartmentSummary AS
(
SELECT
Department,
COUNT(*) TotalEmployees
FROM Employee
GROUP BY Department
)

SELECT *
FROM DepartmentSummary
WHERE TotalEmployees>2;
```

# Example 3 – CTE with JOIN

```sql
WITH EmployeeDepartment AS
(
SELECT
e.Name,
d.Department
FROM Employee e
INNER JOIN Department d
ON e.Dept_ID=d.Dept_ID
)

SELECT *
FROM EmployeeDepartment;
```

Instead of writing the JOIN repeatedly, we create it once.

# Example 4 – CTE with Window Function

```sql
WITH RankedEmployees AS
(
SELECT
Name,
Salary,
ROW_NUMBER() OVER(ORDER BY Salary DESC) RankNo
FROM Employee
)

SELECT *
FROM RankedEmployees
WHERE RankNo<=3;
```

This is a very common pattern in Data Engineering.

- Give meaningful names to CTEs (`HighSalaryEmployees`, `SalesSummary`).
- Keep each CTE focused on a single transformation.
- Chain multiple CTEs to model ETL steps.

- Use comments to describe complex logic.

# 🎯 Summary

- A **CTE (Common Table Expression)** is a temporary named result set created using the `WITH` clause.
- It exists only during the execution of a single SQL statement.
- CTEs improve readability, modularity, and maintainability.
- They are especially useful for breaking complex queries into logical steps.
- Multiple CTEs can be chained together to model real-world ETL pipelines.
- In Data Engineering, CTEs are commonly used with joins, aggregations, window functions, and multi-stage transformations to produce clean, production-ready SQL.

# 🟠 CTE vs Subquery vs Temporary Table vs View

One of the most common SQL interview questions for Data Engineers is:

> **When should I use a CTE, a Subquery, a Temporary Table, or a View?**

Although these features may seem similar because they all help simplify SQL queries, each has a different purpose, scope, lifetime, and performance characteristics.

Choosing the right one depends on your use case.

---

# 1️⃣ CTE vs Subquery

## What is a Subquery?

A **Subquery** (Nested Query) is a query written inside another SQL query.

```sql
SELECT *
FROM Employee
WHERE Salary >
(
SELECT AVG(Salary)
FROM Employee
);
```

---

## Same Query using CTE

```sql
WITH AverageSalary AS
(
SELECT AVG(Salary) AvgSalary
FROM Employee
)

SELECT *
FROM Employee
WHERE Salary >
(
SELECT AvgSalary
FROM AverageSalary
);
```

---

## Comparison

### Subquery

```
Main Query

↓

Subquery

↓

Result
```

Everything is written together.

---

### CTE

```
WITH

↓

Temporary Table Created

↓

Main Query Uses It
```

The logic is separated into readable blocks.

---

## Example

### Without CTE

```sql
SELECT *
FROM
(
SELECT
Department,
AVG(Salary) AvgSalary
FROM Employee
GROUP BY Department
) A

WHERE AvgSalary>60000;
```

---

### With CTE

```sql
WITH DepartmentSalary AS
(
SELECT
Department,
AVG(Salary) AvgSalary
FROM Employee
GROUP BY Department
)

SELECT *
FROM DepartmentSalary
WHERE AvgSalary>60000;
```

Much easier to read.

---

## CTE vs Subquery

| Feature | CTE | Subquery |
|----------|-----|----------|
|Readability|⭐⭐⭐⭐⭐|⭐⭐|
|Reusable|Yes (within same query)|Usually No|
|Debugging|Easy|Hard|
|Nested Logic|Simple|Can become messy|
|Scope|Single Statement|Single Statement|
|Best For|Complex Queries|Simple Queries|

---

## When to Use Subquery

✅ Simple filtering

✅ Single calculation

✅ EXISTS

✅ IN

✅ Small queries

---

## When to Use CTE

✅ ETL Pipelines

✅ Reports

✅ Analytics

✅ Multiple Transformations

✅ Window Functions

---

# 2️⃣ CTE vs Temporary Table

Both create temporary data.

But they work differently.

---

## Temporary Table

A Temporary Table is physically created inside the database temp storage.

```sql
CREATE TABLE #HighSalary
(
ID INT,
Salary INT
);
```

Insert Data

```sql
INSERT INTO #HighSalary

SELECT *
FROM Employee
WHERE Salary>60000;
```

Use

```sql
SELECT *
FROM #HighSalary;
```

Drop

```sql
DROP TABLE #HighSalary;
```

---

## CTE

```sql
WITH HighSalary AS
(
SELECT *
FROM Employee
WHERE Salary>60000
)

SELECT *
FROM HighSalary;
```

No table creation required.

---

## Execution Flow

### Temporary Table

```
Create Table

↓

Insert Data

↓

Use Multiple Times

↓

Drop Table
```

---

### CTE

```
WITH

↓

Temporary Result

↓

Main Query

↓

Deleted Automatically
```

---

## CTE vs Temporary Table

| Feature | CTE | Temporary Table |
|----------|-----|-----------------|
|Storage|Memory/Optimizer Managed|TempDB / Temporary Storage|
|Lifetime|One Query|Entire Session (or until dropped)|
|Reusable|Same Query Only|Multiple Queries|
|Indexes|No|Yes|
|Statistics|Limited/Optimizer-dependent|Yes|
|Performance|Good for small transformations|Better for large reusable datasets|
|Manual Drop|No|Yes (recommended)|

---

## When to Use Temporary Tables

- Large datasets
- Multiple SQL statements
- Repeated calculations
- Intermediate ETL stages
- Complex stored procedures

---

## When to Use CTE

- Single query
- Better readability
- Simple transformations
- Window functions
- Reporting

---

# 3️⃣ CTE vs View

Many beginners think CTE and View are the same.

They are not.

---

## View

A View is a **stored SQL query** inside the database.

```sql
CREATE VIEW EmployeeView AS

SELECT *
FROM Employee;
```

Now use it.

```sql
SELECT *
FROM EmployeeView;
```

The View remains in the database until dropped.

---

## CTE

```sql
WITH EmployeeView AS
(
SELECT *
FROM Employee
)

SELECT *
FROM EmployeeView;
```

The CTE disappears immediately after execution.

---

## CTE vs View

| Feature | CTE | View |
|----------|------|------|
|Permanent|No|Yes|
|Storage|No|Stores SQL definition (not data)|
|Reusable|One Query|Many Queries|
|Create Once|No|Yes|
|Share with Others|No|Yes|
|Database Object|No|Yes|

---

## When to Use View

- Dashboards
- BI Reports
- Security
- Hide Complex SQL
- Reusable Queries

---

## When to Use CTE

- Temporary calculations
- ETL
- One-time analysis
- Complex SQL

---

# 4️⃣ Recursive vs Non-Recursive CTE

There are two types of CTEs.

```
CTE

│

├──────────────┐

│              │

Non Recursive  Recursive
```

---

# Non-Recursive CTE

Runs only once.

```sql
WITH HighSalary AS
(
SELECT *
FROM Employee
WHERE Salary>60000
)

SELECT *
FROM HighSalary;
```

Execution

```
CTE

↓

Result

↓

Finish
```

---

# Recursive CTE

A Recursive CTE calls itself repeatedly until a stopping condition is met.

Used for

- Employee Hierarchy
- Folder Structure
- Organization Tree
- Bill of Materials
- Family Tree
- Graph Traversal

Example

```sql
WITH RecursiveCTE AS
(
Anchor Query

UNION ALL

Recursive Query
)

SELECT *
FROM RecursiveCTE;
```

Execution

```
Anchor Query

↓

Result

↓

Recursive Query

↓

New Result

↓

Recursive Query

↓

New Result

↓

Stop Condition

↓

Finish
```

---

## Example

Employee Table

|EmpID|ManagerID|
|-----|----------|
|1|NULL|
|2|1|
|3|2|
|4|2|

Recursive CTE

```
CEO

↓

Manager

↓

Team Lead

↓

```

(We'll cover Recursive CTEs in detail in the next chapter.)

| Feature | Non-Recursive | Recursive |

|Calls Itself|No|Yes|
|Execution|Once|Repeated|
|Hierarchy|No|Yes|
|Trees|No|Yes|
|Graphs|No|Yes|
|Performance|Faster|Can be slower|

---

### ✅ Use Temporary Tables for reusable intermediate results

If the same dataset is used across multiple SQL statements, a temporary table is often a better fit than a CTE.

---

### ✅ Create Views for business users

Instead of exposing long SQL queries, create reusable views for reporting tools.

---

### ✅ Prefer readability over cleverness

Write SQL that another engineer can understand six months later.

---

# ❌ Common Mistakes

- Using a CTE when data must be reused across multiple statements.
- Assuming a CTE stores data permanently.
- Replacing every subquery with a CTE, even for simple logic.
- Creating very long CTE chains without clear naming.
- Forgetting that recursive CTEs require a termination condition.
- Assuming a View stores data (a standard view stores only the query definition).

# 🎯 Summary

- **CTEs** improve readability and simplify complex SQL logic within a single statement.
- **Subqueries** are ideal for simple nested operations.
- **Temporary Tables** are useful when intermediate data must be reused or optimized.
- **Views** provide reusable, centralized query definitions for reporting and security.
- **Recursive CTEs** are designed for hierarchical and recursive data structures, while **Non-Recursive CTEs** handle one-time transformations.
- Choosing the right approach depends on readability, reusability, performance, and the overall business requirement.

# 📖 Why are CTEs Important in Data Engineering?

In Data Engineering, SQL queries are rarely simple.

Instead of writing queries like

```sql
SELECT * FROM Employee;
```

you usually perform multiple transformations before loading data into a Data Warehouse.

Typical ETL Flow

```
Raw Data

↓

Remove NULL Values

↓

Remove Duplicate Records

↓

Join Multiple Tables

↓

Calculate Metrics

↓

Aggregate Data

↓

Create Final Dataset

↓

Load into Gold Layer
```

Instead of writing one huge SQL query, Data Engineers divide every transformation into multiple CTEs.

Each CTE represents **one ETL step**.

This makes pipelines:

- Easy to understand
- Easy to debug
- Easy to maintain
- Easier for other engineers to review

---

# Example 1 — Customer ETL Pipeline

Suppose we have three tables.

## Customer

|CustomerID|Name|
|-----------|------|
|1|Shaukat|
|2|Rahul|
|3|Aman|

---

## Orders

|OrderID|CustomerID|Amount|
|--------|----------|------|
|101|1|2500|
|102|1|1500|
|103|2|4000|

---

## Payments

|PaymentID|OrderID|Status|
|----------|-------|------|
|1|101|Success|
|2|102|Pending|
|3|103|Success|

---

Business Requirement

> Show total successful order amount for every customer.

---

## Solution using Multiple CTEs

```sql
WITH CustomerOrders AS
(
SELECT
c.CustomerID,
c.Name,
o.OrderID,
o.Amount
FROM Customer c
JOIN Orders o
ON c.CustomerID=o.CustomerID
),

SuccessfulPayments AS
(
SELECT
co.*
FROM CustomerOrders co
JOIN Payments p
ON co.OrderID=p.OrderID
WHERE p.Status='Success'
),

CustomerSummary AS
(
SELECT
Name,
SUM(Amount) TotalAmount
FROM SuccessfulPayments
GROUP BY Name
)

SELECT *
FROM CustomerSummary;
```

---

Execution Flow

```
Customer

+

Orders

↓

CustomerOrders

+

Payments

↓

SuccessfulPayments

↓

Aggregation

↓

CustomerSummary

↓

Output
```

---

Output

|Customer|TotalAmount|
|----------|-----------|
|Shaukat|2500|
|Rahul|4000|

---

Notice how each CTE performs exactly one transformation.

This follows the **Single Responsibility Principle**.

# ⚡ Performance Considerations

Many beginners believe

> **CTE is always faster.**

❌ Wrong.

A CTE is mainly a **readability and maintainability feature**. Whether it improves performance depends on the database optimizer.

Performance depends on

- Indexes
- Statistics
- Joins
- Data Volume
- Execution Plan
- Database Engine

---

## CTE is NOT

- Cache
- Physical Table
- Permanent Storage

It is simply a logical query expression.

---

## When CTE Performs Well

## When CTE May Not Be Ideal

Suppose

```
100 Million Rows
```

Same CTE used

```
10 Times
```

The optimizer may re-evaluate the logic multiple times (database-dependent).

Better solution

```
Temporary Table
```

---

# Example

Instead of

```sql
WITH Sales AS(...)
```

Use

```sql
CREATE TEMP TABLE Sales AS

SELECT...
```

if the result will be reused across multiple statements.

# 🚀 Query Optimization Tips

Optimization is one of the most important skills for a Data Engineer.

---

## 1. Keep CTE Small

Good

```
Clean Data
```

Bad

```
500-line CTE
```

---

## 2. Filter Early

Bad

```sql
SELECT *

FROM Orders
```

Good

```sql
SELECT *

FROM Orders

WHERE OrderDate>='2025-01-01'
```

Reducing rows early minimizes downstream work.

---

## 3. Select Required Columns

Bad

```sql
SELECT *
```

Good

```sql
SELECT

CustomerID,

Amount
```

Less data = faster execution.

---

## 4. Avoid Duplicate Calculations

Bad

```
Calculate SUM

Again

Again

Again
```

Good

```
Calculate Once

Reuse
```

---

## 5. Use Meaningful CTE Names

Bad

```sql
WITH A AS(...)
```

Good

```sql
WITH CustomerSales AS(...)
```

---

## 6. Avoid Too Many Nested CTEs

```
CTE1

↓

CTE2

↓

CTE3

↓

CTE4

↓

CTE5

↓

CTE6
```

Keep pipelines understandable. If transformations become too large or need reuse across multiple statements, consider temporary tables.

---

## 7. Use Proper Indexes on Source Tables

CTEs don't have indexes.

The optimizer relies on indexes from the underlying tables.

---

## 8. Push Filters Before JOIN

Good

```
Filter

↓

Join
```

Bad

```
Join

↓

Filter
```

Filtering first reduces the amount of data participating in joins.

---

# ⭐ Best Optimization Strategy

```
Read Less Data

↓

Filter Early

↓

Join Less

↓

Aggregate

↓

Return Only Required Columns
```

---

# 🎯 Key Takeaways

- CTEs are excellent for organizing ETL logic into clear, sequential steps.
- Each CTE should ideally perform one transformation, making pipelines easier to maintain.
- CTEs are heavily used in medallion architectures, reporting, analytics, and data quality workflows.
- Performance is driven primarily by the database optimizer, indexes, query design, and data volume—not simply by using a CTE.
- Apply optimization techniques such as early filtering, selecting only required columns, meaningful naming, and avoiding unnecessary complexity to build efficient, production-ready SQL.
